# TIL: EC2 스토리지와 IAM 권한 체계 (2026-07-04)

AWS DevOps 엔지니어 교육 7기 클라우드 기초 실습 1을 진행하며 배운 내용 정리.

## 1. EC2 Instance Store vs EBS

### 개념
| | Instance Store | EBS |
|---|---|---|
| 정체 | 물리 서버에 직접 붙은 로컬 디스크 | 네트워크로 연결된 별도 스토리지 |
| Reboot | 데이터 유지 | 데이터 유지 |
| Stop → Start | **데이터 소실 가능** (다른 물리 호스트로 재배치될 수 있음) | 데이터 유지 |
| 속도 | 빠름 (로컬 NVMe) | 상대적으로 느림 (네트워크 경유) |
| 용도 | 캐시, 임시 데이터 | DB, 영구 보존 데이터 |

### 실습 중 발견한 함정
- 실습 자료는 구형 `c3.large`(Xen 가상화) 기준이었는데, 지금은 단종되어 `c5d.large`(Nitro 기반)로 대체함
- **구형(Xen) 세대는 AMI가 인스턴스 스토어를 자동으로 `/mnt`에 마운트**해줬지만, **신형(Nitro) 세대는 수동 마운트가 필요**함
- `/mnt`에 파일을 만들고 Stop→Start 했는데 파일이 그대로 남아있어서 원인을 조사함
- `lsblk`, `df -h`로 확인한 결과, 진짜 인스턴스 스토어 디스크(`/dev/nvme1n1`)는 마운트조차 안 되어 있었고, `/mnt`는 그냥 루트 볼륨(EBS) 안의 평범한 폴더였음
- → 실무에서도 흔히 발생하는 이슈: "인스턴스 스토어를 썼는데 데이터가 안 보인다" = 마운트 누락

```bash
# 인스턴스 스토어 디스크 확인
lsblk
df -h

# 진짜로 마운트하려면
sudo mkfs -t xfs /dev/nvme1n1
sudo mount /dev/nvme1n1 /mnt
```

## 2. IAM 역할(Role)로 EC2에 권한 부여

### 왜 필요한가
EC2 안에서 `aws s3 ls` 같은 명령으로 다른 AWS 서비스를 호출하려면 인증이 필요함.
- ❌ 나쁜 방법: 액세스 키를 인스턴스에 하드코딩 (유출 위험, 키 순환 번거로움)
- ✅ 좋은 방법: IAM Role을 인스턴스에 연결 → AWS가 임시 자격증명을 자동 발급/갱신

### 실습 순서
1. 역할 없이 `aws s3 ls` 실행 → `Unable to locate credentials` 에러 확인
2. IAM 콘솔에서 역할 생성
   - 신뢰할 수 있는 엔터티: AWS 서비스 → EC2
   - 정책: `AmazonS3ReadOnlyAccess` (최소 권한 원칙)
3. EC2 → 작업 → 보안 → IAM 역할 수정 → 역할 연결
4. 재부팅 없이 바로 `aws s3 ls` 재실행 → 성공

**포인트**: 역할을 붙이는 순간 바로 적용됨 (인스턴스 재시작 불필요)

## 3. IAM 사용자 & 사용자 그룹

- **그룹은 사람(User)만 소속 가능**, Role은 그룹에 넣을 수 없음
- 그룹을 여러 개 만들어서 직무별로 권한을 분리하는 게 실무 표준 패턴

- **한 사용자가 여러 그룹에 동시 소속 가능** → 권한이 합쳐짐 (예: 개발 권한 + 비용 조회 권한)

### 실습
- `iam-pyc` 사용자 생성 (콘솔 로그인 가능, 최초 로그인 시 비밀번호 변경 강제)
- `group-pyc` 그룹 생성 → `AmazonS3FullAccess` 정책 연결 → `iam-pyc`를 그룹에 소속

## 4. Role vs User/Profile vs Policy — 용어 정리

| 용어 | 정체 | 자격증명 방식 |
|---|---|---|
| **Role** | 서비스(EC2 등)나 다른 계정이 "빌려 쓰는" 임시 권한 그릇 | 고정 키 없음, 임시 토큰 자동 발급/갱신 |
| **User** | 사람에게 부여되는 고유 계정 | 고정된 Access Key ID + Secret Key |
| **Group** | User들의 묶음 (역할별 권한 관리용) | - |
| **Policy** | 권한의 실제 내용 (JSON 문서) | User/Group/Role 어디든 붙일 수 있음 |

**정리**: User/Group/Role은 권한을 담는 "그릇"이고, Policy는 그 안에 담기는 "내용물".

- Role은 그릇이지만 그룹에는 못 들어감 (그룹 = 오직 User들의 묶음)
- Role과 Profile(User의 액세스 키)은 서로 완전히 독립적인 인증 경로

## 5. AWS CLI Profile

### 왜 필요한가
한 사람이 여러 자격증명(회사 계정, 개인 계정 등)을 오갈 때 구분해서 관리하기 위함.

```bash
aws configure --profile tester
# Access Key ID, Secret Key, Region, Output format 입력

aws --profile tester s3 ls
```

### 실습으로 확인한 핵심
- 인스턴스 Role은 `EC2RoleForSSM` (S3 권한 없음)이었는데도
- `--profile tester`를 붙이니 `iam-pyc` 사용자(S3FullAccess)의 권한으로 인증되어 성공
- → **인스턴스 Role**과 **CLI Profile(User 자격증명)**은 서로 다른 층(layer)의 인증이며, 독립적으로 공존함
- 실무 예: 서버엔 최소 권한 Role만 붙여두고, 개발자가 접속했을 때 자기 개인 Profile로 추가 작업 가능

## 6. Session Manager (실습은 계정 설정 문제로 미완료)

### 개념
- SSH 접속: 키 페어 관리 필요, 22번 포트 개방 필요
- Session Manager: IAM 권한(`AmazonSSMManagedInstanceCore`)만 있으면 브라우저에서 바로 접속, 포트 개방 불필요

### 막힌 이유
계정 전체 차원의 SSM Default Host Management 설정이 꺼져있어서 발생. 학생 권한으로는 해결 불가, 계정 관리자 설정이 필요한 부분.
