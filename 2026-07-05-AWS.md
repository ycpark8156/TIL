## 1. AWS CLI + MFA (시도, 개념 정리로 마무리)

### 개념
- 콘솔 로그인 MFA와 **CLI 세션 MFA는 완전히 별개**임

## 2. IAM 정책 평가 — 명시적 거부(Explicit Deny) 우선 원칙

### 핵심 규칙
1. 관련된 모든 정책(그룹 + 사용자 직접 + 역할)을 모두 취합
2. **하나라도 명시적 Deny가 있으면 무조건 거부** (Allow가 아무리 많아도 소용없음)
3. Deny가 없고 Allow가 있으면 허용
4. 아무 것도 없으면 기본은 거부 (Implicit Deny)

### 실습
- S3 버킷(`policy-test-student02`) 생성 후 업로드 성공 확인 (Before 상태)
- `iam-pyc` 사용자에게 인라인 정책 `DenyS3Write` 추가 (`s3:PutObject`만 명시적 거부)
- 그룹(`group-pyc`)을 통한 `AmazonS3FullAccess`는 여전히 살아있지만, 개인 Deny가 우선 적용된다는 원리를 개념으로 확인


### Role vs User/Profile vs Policy vs Group 용어 정리
| 용어 | 정체 | 그룹 소속 가능? |
|---|---|---|
| Role | 서비스(EC2, Lambda 등)가 빌려 쓰는 임시 권한, 고정 키 없음 | ❌ 불가능 (그룹은 User 전용) |
| User | 사람에게 부여된 고유 계정, 고정 Access Key | ✅ |
| Group | User들의 묶음, 직무별 권한 관리용 | - |
| Policy | 권한의 실제 내용(JSON), User/Group/Role 어디든 부착 가능 | - |

- 한 User는 여러 Group에 동시 소속 가능 → 권한이 합쳐짐 (예: 개발 권한 + 비용 조회 권한 그룹을 동시에)

## 3. Security Group (인스턴스 단위 방화벽)

- **적용 단위**: 인스턴스(정확히는 ENI)
- **Stateful**: 인바운드 허용하면 그 응답 아웃바운드는 자동 허용됨
- 기본 상태: 인바운드 규칙 0개(전체 차단), 아웃바운드는 "모든 트래픽 허용" 1개 기본 포함
- **Allow만 가능, Deny 규칙 자체가 없음** — 규칙에 없으면 자동으로 막히는 것이지, 명시적으로 거부하는 게 아님

### 실습
- `private-instance-pyc` SG 생성
- SSH(22) 인바운드를 처음엔 `0.0.0.0/0`으로 열었다가, **"내 IP"**로 좁혀서 특정 IP(`119.193.240.207/32`)에서만 접속 가능하도록 수정
- 아웃바운드도 기본 전체 허용 규칙을 지우고 필요한 포트만 남기면 제한 가능하다는 것 확인 (보안 사고 시 외부 유출 방지 목적)

## 4. NACL (서브넷 단위 방화벽)

| | Security Group | NACL |
|---|---|---|
| 적용 위치 | 인스턴스(ENI) | 서브넷 전체 |
| 상태 | Stateful | **Stateless** (인바운드/아웃바운드 각각 규칙 필요) |
| 규칙 | Allow만 | **Allow + Deny 둘 다 가능** |
| 평가 방식 | 전체 규칙 종합 판단 | **규칙 번호 순서대로** 먼저 매칭되는 것 적용 |

- 트래픽 흐름: 인터넷 → **서브넷의 NACL** 통과 → **인스턴스의 SG** 통과 → 인스턴스 도달 (둘 다 통과해야 함)
- 특정 IP만 콕 집어 차단하고 싶을 때는 SG로는 불가능 → NACL의 Deny 규칙이 필요

## 5. Lambda + EventBridge로 EC2 자동 시작 (실제 동작 확인 완료)

### 구조
EventBridge (예약 실행, cron)
↓
Lambda 함수 (boto3로 EC2 API 호출)
↓
EC2 인스턴스 (자동 시작/중지)
### 실습 순서
1. `lambda-ec2-role` 생성: 신뢰 대상 AWS 서비스 → Lambda, 정책 `AmazonEC2FullAccess`
2. Lambda 함수 `start-ec2-instances-pyc` 생성 (Python), 실행 역할로 위 Role 연결
3. 코드 작성: `auto-start=true` 태그가 붙은 인스턴스를 찾아 `start_instances()` 호출

```python
import boto3
region = "ap-southeast-3"
ec2 = boto3.client('ec2', region_name=region)

response = ec2.describe_tags(
    Filters=[{'Name': 'tag:auto-start', 'Values': ['true']}]
)
instance_ids = [x["ResourceId"] for x in response["Tags"] if x["ResourceType"] == "instance"]

def lambda_handler(event, context):
    result = ec2.start_instances(InstanceIds=instance_ids)
    return result
```

4. 테스트용 인스턴스(`lambda-test-pyc`)를 중지 상태로 만들고 `auto-start=true` 태그 부착
5. Lambda 테스트 실행 → 로그에서 `stopped(80)` → `pending(0)` 상태 전환 확인
6. EC2 콘솔에서 실제로 인스턴스가 **"실행 중"** 으로 바뀐 것 최종 확인 완료 


## 오늘의 핵심
1. 콘솔 로그인 MFA와 CLI 세션 MFA는 서로 다른 층의 보안이며, CLI profile은 인스턴스 로컬 파일이라 인스턴스가 바뀌면 재설정이 필요하다
2. IAM은 Deny가 항상 Allow를 이긴다 — 그룹 정책을 안 건드리고도 개인 Deny로 예외 처리가 가능하다
3. Security Group(인스턴스 단위, Allow만) vs NACL(서브넷 단위, Allow+Deny, Stateless)의 차이와 트래픽이 두 계층을 순서대로 통과해야 한다는 점
4. Lambda + boto3로 태그 기반 EC2 자동 시작/중지가 실제로 동작하는 것을 확인 — EventBridge 스케줄과 결합하면 비용 절감용 자동화가 완성됨
