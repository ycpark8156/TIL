## 2. CIDR 표기법

**개념**
- IP는 32비트, `/N`은 "앞 N비트는 고정, 나머지는 자유롭게 쓸 수 있는 호스트 영역"이라는 뜻
- 숫자가 작을수록 고정 부분이 적어져서 사용 가능 IP 범위가 넓어짐

| CIDR | 고정 비트 | 사용 가능 IP | 비유 |
|------|-----------|--------------|------|
| /16  | 앞 16비트 | 65,536개     | 도시 하나 규모 |
| /24  | 앞 24비트 | 256개        | 동네 하나 규모 |

VPC 전체를 `/16`(큰 땅)으로 만들고, 그 안 서브넷들을 `/24`(작은 구역) 단위로 쪼개 쓰는 구조

## 3. VPC 네트워킹을 마법사 없이 "VPC만" 옵션으로 직접 구성

**목적**: Terraform으로 나중에 코드화할 때 리소스 간 연결 관계를 이해하기 위해, 자동 마법사 대신 단계별로 직접 구성

**실습 순서**
1. VPC 생성 (`192.168.0.0/16`)
2. 서브넷 2개 생성, 가용영역 다르게 (`192.168.11.0/24`, `192.168.12.0/24`)
3. 인터넷 게이트웨이(IGW) 생성 → VPC에 연결(Attach)
4. 라우트 테이블에 `0.0.0.0/0` → IGW 라우트 추가
5. 서브넷 2개를 이 라우트 테이블에 명시적으로 연결

**핵심 개념**
- VPC 자체는 완전히 고립된 사설망 → IGW 없이는 인터넷과 전혀 통신 불가
- IGW를 VPC에 붙여도, 라우트 테이블에 `0.0.0.0/0` → IGW 규칙이 없으면 무용지물
- "퍼블릭 서브넷"은 실체가 있는 속성이 아니라, 그 서브넷이 연결된 라우트 테이블에 IGW로 가는 라우트가 있는지 여부로 결정되는 것

## 4. ALB(Application Load Balancer) + Target Group 구조

**개념**
- **Target Group**: ALB가 트래픽을 보낼 대상(EC2 인스턴스) 명단. ALB에 연결되기 전까지는 헬스체크 자체가 시작 안 됨 (상태: "Unused")
- **ALB의 역할**: SSL/TLS 종료(termination) — HTTPS 암호화를 ALB가 처리하고, 뒤의 EC2/GitLab은 HTTP로만 응답하면 됨. 인증서를 서버마다 관리할 필요 없이 ALB 한 곳에만 붙이면 됨
- **로드밸런싱**: 서버가 여러 대로 늘어나도 ALB가 트래픽을 알아서 분산

**실습**
- Target Group 생성: HTTP:80, 상태검사 경로 `/`, 성공 코드 `200,301,302` (GitLab이 미인증 상태에서 302 리다이렉트를 주므로 이것도 정상으로 인식하게 설정)
- ALB 생성: 인터넷 경계, 가용영역 2개 매핑, 전용 보안그룹(HTTP/HTTPS 인바운드 위치무관)
- HTTP:80 리스너 → Target Group 연결

## 5. GitLab external_url과 HTTPS 강제 리다이렉트 트러블슈팅

**증상**: `docker-compose.yml`에서 `external_url`을 `https://...`로 설정하니, HTTP로 접속 시 브라우저에서 "연결 거부(ERR_CONNECTION_REFUSED)" 발생

**원인 분석**
- `curl -I http://<주소>` 결과 `302 Found` + `Location: https://...` 확인 → GitLab 서버 자체는 정상 응답 중
- GitLab은 `external_url`이 https면 HTTP 요청을 자동으로 HTTPS로 리다이렉트함
- 문제는 그 시점에 ALB에 HTTPS(443) 리스너가 아직 없어서, 리다이렉트된 요청을 받아줄 곳이 없었던 것

**해결**
1. ACM에서 인증서 요청 (도메인: `ycpark.manoit.co.kr`, DNS 검증)
2. Route53에 검증용 CNAME 레코드 추가 ("Route53에서 레코드 생성" 버튼으로 자동 생성)
3. 인증서 발급(Issued) 확인 후 ALB에 HTTPS:443 리스너 추가, 인증서 연결
4. Route53에 최종 CNAME 레코드 추가: `ycpark.manoit.co.kr` → ALB DNS 이름

**최종 아키텍처**
사용자 → https://ycpark.manoit.co.kr
↓ (Route53: 도메인 → ALB 주소 변환)
ALB (HTTPS:443 수신, SSL 종료)
↓ (내부 HTTP:80 전달)
EC2 → GitLab 컨테이너 (nginx['listen_https'] = false)

## 오늘의 핵심

- VPC를 마법사 없이 직접 구성하면서 IGW → 라우트 테이블 → 서브넷 연결이라는 3단계가 있어야 서브넷이 진짜 "퍼블릭"이 된다는 것을 실습으로 확인
- ALB는 로드밸런싱뿐 아니라 SSL 종료(termination) 역할이 크다 — 인증서 관리를 ALB 한 곳으로 집중시켜 백엔드 서버(GitLab)는 HTTP만 신경 쓰면 되게 만드는 구조
- 에러 메시지(연결 거부)만 보고 판단하지 말고 `curl -I`로 서버 응답을 직접 확인해서, 문제가 "서버"가 아니라 "그 앞단(ALB 리스너)"에 있다는 걸 정확히 진단하는 과정이 중요했음
- 인증서(ACM)는 도메인 소유 검증이 필수이며, DNS 검증 방식이 Route53과 연동되면 매우 빠르고 정확하게(자동 레코드 생성 기능) 처리됨
